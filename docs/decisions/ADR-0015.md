# ADR: Radar del Tribunal como Premium Gate — Frecuencias INAP como Ventaja Competitiva

**ID**: ADR-0015
**Fecha**: 2026-03-02
**Estado**: ✅ Aceptado
**Autor**: Claude / Aritz

## 1. Contexto y Problema

OPTEK necesita una feature que:
1. Justifique el precio Premium frente a alternativas como OpositaTest (€95/año).
2. Sea técnicamente reproducible con los datos ya ingestados (preguntas_oficiales + legislacion).
3. Ofrezca un valor no disponible en ningún competidor del mercado español.

La pregunta es: **¿cómo monetizar los datos de exámenes reales sin revelarlos al completo?**

## 2. Opciones Consideradas

### Opción A — Revelar exámenes históricos completos (gratis)
- Pro: Adquisición de usuarios. Con: Destruye el moat — cualquier competidor puede replicar.
- **Descartada**.

### Opción B — Radar del Tribunal como feature Premium (elegida)
- Analizar qué artículos aparecen más en exámenes INAP → ranking de frecuencias.
- Top 20 visible para todos (lead magnet), 21+ bloqueado para usuarios free.
- Botón "Practicar con los más frecuentes" genera un test forzado con esos artículos.
- **Elegida** — valor real, reproducible con datos existentes, diferencial claro.

### Opción C — Exámenes simulacros como único Premium gate
- Solo bloquear los simulacros oficiales para usuarios de pago.
- Pro: Simple. Con: Insuficiente como justificación de precio.
- **Descartada como gate único** (los simulacros se mantienen como feature separada).

## 3. Decisión Elegida: Radar del Tribunal — Opción B

### Implementación técnica

**Pipeline (determinista, sin IA):**
1. `execution/build-radar-tribunal.ts` — script idempotente que lee `preguntas_oficiales`, extrae citas con `extractCitations()` (regex determinista), y hace UPSERT en `frecuencias_articulos`.
2. `frecuencias_articulos` — tabla con `legislacion_id PK, num_apariciones, pct_total, anios[], ultima_aparicion`.
3. `radar_tribunal_view` — VIEW que hace JOIN con `legislacion` y ordena por frecuencia DESC.
4. Cron `pnpm build:radar` ejecutado tras cada ingesta de examen nuevo.

**Acceso tiered:**
- Free: top 20 artículos (tabla con blur+lock a partir de posición 21)
- Premium: top 100 artículos completo + botón "Generar test Radar" (`tipo='radar'` en la API)
- El test radar usa `generateTopFrecuentesTest()` — contexto forzado con los top artículos, sin RAG por tema

### Por qué este diseño específico

**WHY frecuencias y no preguntas crudas:** Mostrar las preguntas completas de los exámenes INAP viola potencialmente los derechos de reproducción del INAP. Mostrar *estadísticas de frecuencia* derivadas (cuántas veces aparece el Art. 21 LPAC) es legalmente sólido — es análisis, no reproducción.

**WHY extractCitations() para el análisis:** El mismo sistema de verificación determinista que usamos para validar outputs de IA es reutilizado aquí en sentido inverso — para *extraer* qué artículos pregunta el tribunal. Coherencia arquitectónica: una sola lógica de parsing de citas para todo el sistema.

**WHY top 20 gratis (y no 0 o 10):** El paywall debe mostrar suficiente valor para generar curiosidad ("¿qué hay más allá del puesto 20?") sin regalar el insight completo. 20 artículos = ~30% del examen típico de 60 preguntas — suficiente para convencer, insuficiente para sustituir el Premium.

**WHY tipo='radar' sin tema_id:** El test del Radar es cross-tema por definición (mezcla LPAC, CE, TREBEP según frecuencia real). Forzar un tema_id sería artificial. La BD acepta `tema_id = NULL` para tests especiales (psicotécnicos, flash tests, radar).

## 4. Consecuencias

### Positivas
- **Moat de datos:** Los datos de exámenes históricos son difíciles de obtener — el INAP no los publica de forma estructurada. OPTEK los tiene ingestados y analizados.
- **Coste €0 adicional:** El Radar no consume tokens de IA. Solo SQL + regex determinista.
- **Diferencial claro frente a competidores:** OpositaTest, MasterD, CEF no ofrecen análisis de frecuencia de artículos. Es un argumento de venta único.
- **Expansión natural:** Al añadir exámenes de C1 (Administrativo) o A2 (Gestión), el radar se actualiza con `pnpm build:radar` — sin cambios de código.

### Negativas / Riesgos
- **Dependencia de calidad de parseo:** Si `extractCitations()` no extrae citas de algunos formatos de pregunta, el ranking puede estar incompleto. Mitigación: los tests unitarios en `tests/unit/build-radar-tribunal.test.ts` validan el pipeline.
- **Cobertura limitada de citas:** Las preguntas que no citan artículos explícitamente (preguntas de concepto general) no contribuyen al radar. Esto es una limitación inherente, no un bug.
- **Exámenes 2018 sin parsear:** Los PDFs de 2018 son escaneados y requieren OCR. Hasta que se procesen, el radar tiene datos de 2019, 2022, 2024 únicamente.

## 5. Métricas de éxito

- `frecuencias_articulos` con ≥30 artículos tras ejecutar `pnpm build:radar` con los exámenes ingestados.
- Top 1 del radar coincide con artículos conocidos de alta frecuencia en el INAP (Art. 21 LPAC o similar).
- Quality Score del test generado con `tipo='radar'` ≥ 85% (mismo umbral que tests normales).
